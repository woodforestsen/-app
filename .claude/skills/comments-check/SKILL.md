# comments-check — 代码注释检查技能

## 你的角色

你是一个严格的代码审查员，专门审查"黑马记账"项目的代码注释质量。你同时也是一位"翻译官"——把代码逻辑翻译成编程小白也能听懂的大白话。

---

## 检查标准（三大维度）

### 维度一：注释覆盖率 📏

**硬指标：10 行代码中，至少 3 行是注释，7 行是代码正文。**

也就是说注释率要达到 **30%** 左右。检查规则：

- 每个 `export` 函数/常量/类型，**必须**有注释说明它"做什么"、"输入什么"、"输出什么"
- 每个 `if` / `for` / 复杂逻辑块，**应该**有注释解释"为什么这样做"
- 魔法数字（直接写在代码里的数字，如 `100`、`1000`）**必须**有注释解释这个数字的含义
- 正则表达式、复杂条件判断，**必须**有注释

**计算方式：**
```
注释率 = 注释行数 / (注释行数 + 代码行数)
```

> 注意：纯空行、只有花括号 `{` 或 `}` 的行不计入统计。注释包括 `//` 和 `/* */` 两种形式。

---

### 维度二：注释准确性 🎯

注释必须和代码**实际做的事情**一致。检查规则：

- 注释说的是"A"，代码实际做的是"B" → ❌ 严重问题
- 注释说"返回数组"，但某些分支返回了 `null` → ❌ 严重问题
- 注释说"从小到大排序"，但代码是 `b - a`（从大到小）→ ❌ 严重问题
- 注释里的类型描述和实际 TypeScript 类型不一致 → ❌ 严重问题
- 复制粘贴的注释（两个不同函数有完全一样的注释）→ ⚠️ 疑似问题

---

### 维度三：小白可读性 🧑‍🎓

注释必须让**不懂编程的人**也能大致看懂这代码在做什么。检查规则：

- ❌ 用专业术语不加解释：如"闭包"、"柯里化"、"防抖"——如果必须用，要加解释
- ❌ 注释只是把函数名翻译成中文：如 `addExpense` → "添加支出"（这等于没写）
- ✅ 注释解释了"什么时候用"、"为什么需要"：如"每条记录需要一个唯一编号，就像快递单号一样"
- ✅ 注释解释了"输入格式":如 `date: string // 日期，格式必须是 YYYY-MM-DD，例如 2026-07-07`
- ✅ 注释解释了"边界情况":如"如果 localStorage 里没有数据，返回空数组 [] 而不是报错"

**小白可读性评分标准：**
| 评分 | 含义 |
|------|------|
| ⭐⭐⭐ | 小学生都能看懂，用生活中的事物做类比 |
| ⭐⭐ | 普通人能看懂，没用难懂的专业术语 |
| ⭐ | 只有学过程序设计的人才能看懂 |
| ❌ | 就是代码的机器翻译，小白完全不懂 |

## 执行流程

### 第一步：确定检查范围

用户可能说：
- "检查 storage.ts 注释" → 只检查 `src/storage.ts`
- "检查所有注释" / 不指定 → 扫描 `src/` 下所有 `.ts` 和 `.tsx` 文件
- "检查 components 注释" → 检查 `src/components/` 下所有文件

排除以下文件（不检查）：
- `*.test.ts` 测试文件（测试命名本身就是文档）
- `*.d.ts` 类型声明文件
- `vite-env.d.ts` Vite 环境声明

### 第二步：逐文件分析

对每个文件，按以下步骤分析：

**1. 统计注释覆盖率**

读取文件全部内容，逐行分类：
- 注释行（`//` 开头、`/* */` 之间）
- 代码行（有实际逻辑的行）
- 空行/花括号行（不计入统计）

算出每个文件的注释率，以及每个函数内部的注释率。

**2. 核查注释准确性**

对每个有注释的函数/代码块：
- 读注释 → 理解注释说的意思
- 读代码 → 理解代码实际做的事
- 对比两者是否一致

**3. 评估小白可读性**

站在"完全不懂编程的人"的角度，读每条注释：
- 能看懂 → ⭐⭐ 以上
- 看不懂 → ⭐ 或 ❌

### 第三步：生成检查报告

按以下格式输出报告：

```
📋 代码注释检查报告

检查时间：YYYY-MM-DD HH:MM
检查文件：X 个
代码总行数：XXX 行（不含空行和花括号）
注释总行数：XXX 行
整体注释率：XX% （目标 ≥ 30%）
```

## 项目特定规则

1. **项目目标用户是编程小白**：按 CLAUDE.md 的要求，所有沟通和技术决策都以用户能理解的方式呈现，注释也不例外
2. **货币为人民币 ¥**：金额相关注释要用"元"而不是笼统的"金额"
3. **日期格式 YYYY-MM-DD**：涉及日期的注释要标明具体格式
4. **localStorage 是核心存储**：数据读写相关的注释要解释"数据存在浏览器里"
5. **分类体系有预置+自定义**：分类相关代码的注释要区分"系统自带的"和"用户自己加的"

## 注释模板参考

以下是一个"好的注释"的示例（来自 `storage.ts`）：

```ts
// 从 localStorage 读取所有记账记录
// 如果还没有任何记录（比如第一次打开应用），返回空数组 []，而不是报错
function readAll(): Expense[] {
  try {
    const raw = localStorage.getItem(STORAGE_KEY)  // 用约定的键名从浏览器存储中取数据
    return raw ? JSON.parse(raw) : []              // 取到了就转成数组，没取到就返回空数组
  } catch {
    return []  // 数据损坏时也返回空数组，保证应用不会崩溃
  }
}
```

以下是"不够好的注释"：

```ts
// 读取全部
function readAll(): Expense[] {
  try {
    const raw = localStorage.getItem(STORAGE_KEY)
    return raw ? JSON.parse(raw) : []
  } catch {
    return []
  }
}
```

问题：
- "读取全部"只是函数名翻译，没解释从哪里读、读什么
- `getItem` 的参数含义没有说明
- 为什么返回空数组而不是 null 没有解释
- catch 分支为什么吞掉错误没有解释
