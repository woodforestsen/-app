# 黑马记账 — 产品文档 & 开发规范

---

## 一、项目概述

**项目名称**：黑马记账
**项目类型**：个人记账桌面应用
**目标平台**：Windows + macOS
**货币单位**：人民币（¥）

---

## 二、核心功能

1. **记录花销**：记录每笔支出的金额、日期、分类、备注（可选）
2. **二级分类体系**：一级大类 → 二级小类，每笔花销归属一个二级小类
3. **花销列表**：按时间倒序展示所有记录，支持按分类筛选
4. **月度统计**：查看每月总支出及各分类占比

## 四、技术栈（已选定）

> **React + TypeScript + Vite + localStorage**（2026-07-06 最终确定）
> 
> 原选 Electron 方案因 Windows 平台已知 bug 改为纯网页方案。未来可随时加回 Electron 壳。

| 层级 | 技术 | 用途 |
|------|------|------|
| 界面框架 | React 18 | 构建用户界面 |
| 编程语言 | TypeScript | 类型安全的 JavaScript |
| 构建工具 | Vite 5 | 极速开发服务器和打包 |
| 数据存储 | localStorage | 浏览器本地存储，无需联网 |
| 样式 | CSS | 界面美化 |

**运行方式**：
- 开发模式：`npm run dev`（浏览器打开 http://localhost:5173）
- 生产打包：`npm run build`（输出到 dist/ 文件夹）
- 预览打包：`npm run preview`

## 五、开发规则（极其重要！）

> **用户是编程小白，本规则在整个项目期间必须严格遵守：**

1. **任何技术决策**（选库、选方案、选架构、选工具等），Claude 必须：
   - 列出 2-3 个可行方案
   - 用通俗语言（类比日常事物）解释每个方案的优劣势
   - 给出明确推荐及理由
   - **等待用户明确选择后**再继续执行
   - 绝不替用户做技术决定

2. **代码修改前**，先解释：
   - 要改什么文件
   - 为什么要改
   - 改完后效果是什么

3. **每完成一个功能模块**，用通俗语言告知：
   - 做了什么
   - 用户如何查看/验证效果

4. **所有沟通以中文为主**，避免使用编程术语而不加解释

5. **遇到错误或问题时**：
   - 不要慌张，用通俗语言解释发生了什么
   - 列出解决方案让用户选择
